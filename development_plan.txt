EXECUTIVE SUMMARY
================================================================================

This document describes the architecture and implementation strategy for an
end-to-end system that (1) ingests a LinkedIn profile, (2) renders it as an
interactive resume in a Python Shiny web application, and (3) learns from
visitor behavior to suggest new visualizations and features over time.

The system has three distinct personas:
  - Resume Visitor: reads the resume, implicitly generates behavioral signals
  - Resume Owner (Admin): imports data, reviews analytics, accepts/rejects
    AI-generated visualization suggestions
  - AI Layer: analyzes behavioral signals and proposes improvements via Claude API

================================================================================
1. CORE TECHNICAL CHALLENGES
================================================================================

1.1 LinkedIn Data Acquisition
-------------------------------
LinkedIn deprecated its public API for third-party profile access in 2015.
This project scrapes the owner's own profile programmatically using the
unofficial internal LinkedIn API — the same JSON endpoints LinkedIn's own
mobile app uses.

  Primary approach — linkedin-api library (pip install linkedin-api)
    - Authenticates with LinkedIn username + password (stored as env vars)
    - Calls LinkedIn's internal REST endpoints directly (no browser required)
    - Returns structured Python dicts for: profile, experience, education,
      skills (with endorsement counts), certifications, recommendations,
      connections, and contact info
    - Lightweight: no browser binary, fast to execute
    - Schedulable: can run on a cron job or triggered from the admin dashboard
    - Risk: technically violates LinkedIn ToS; LinkedIn may issue a CAPTCHA
      challenge or temporary account checkpoint on suspicious activity
    - Mitigations:
        - Run scrapes infrequently (weekly or on-demand, not continuous)
        - Use a residential IP or your home network, not a cloud/datacenter IP
        - Add jitter/delay between API calls (1–3 seconds between requests)
        - Handle auth failures and checkpoints gracefully (see Section 5.1)

  Fallback approach — Playwright browser scraping
    - Full browser automation: navigate to your own LinkedIn profile page
      and extract data from the rendered DOM
    - More realistic traffic pattern; harder for LinkedIn to flag
    - Heavier (requires Chromium binary); more fragile (DOM changes break it)
    - Use only if linkedin-api is blocked or returns incomplete data

  Rejected — LinkedIn Official API (Partner Program)
    - Requires LinkedIn partnership approval; impractical for personal projects

Decision: Use linkedin-api as the primary scraper. Implement Playwright as
an explicit fallback. Treat authentication failure and rate limiting as
first-class failure modes with defined recovery paths (see Section 5.1).

1.2 Behavioral Learning System
--------------------------------
"Learning from user behavior" introduces an ML/feedback loop challenge:

  Signal collection:
    - Which resume sections were viewed and for how long
    - Scroll depth per section
    - Visualization interactions (hover, expand, click, export)
    - Explicit feedback (thumbs up/down on individual visualizations)
    - Referral source and device type

  Signal storage:
    - Events written to DuckDB in near-real-time via a lightweight API
      endpoint embedded in the Shiny app
    - Schema: event_id, session_id (hashed), event_type, section, element,
      timestamp, metadata (JSON)

  Learning pipeline (runs on schedule or on-demand):
    - Aggregate: compute engagement scores per visualization type and section
    - Rank: score = weighted sum of (view_time, interactions, explicit_feedback)
    - Suggest: send top/bottom performers + current viz inventory to Claude API
      with a structured prompt requesting specific improvement suggestions
    - Store suggestions with status: pending / accepted / rejected

  Key challenge — cold start:
    - No behavioral data exists on first deploy
    - Solution: seed with a rule-based suggestion set (see Section 5) that
      activates until sufficient signal is collected (threshold: 50 sessions)

1.3 Visualization Intelligence
--------------------------------
The suggestion engine must produce actionable, renderable suggestions — not
just natural language feedback.

  Output contract for each suggestion:
    {
      "suggestion_id": str,
      "section": str,           # e.g., "skills", "experience"
      "current_viz_type": str,  # e.g., "bar_chart"
      "proposed_viz_type": str, # e.g., "radar_chart"
      "rationale": str,         # human-readable explanation
      "implementation_hint": str, # Python/Shiny pseudo-code for the owner
      "confidence": float,      # 0–1, based on signal volume
      "status": str             # pending | accepted | rejected
    }

  The Claude API call uses the following context:
    - Current visualization inventory (types, sections, creation date)
    - Aggregated engagement metrics per visualization
    - Resume content summary (anonymized section names and data shapes)
    - Prompt: "Given this engagement data, suggest 3 specific visualization
      improvements. Return structured JSON matching the contract above."

1.4 DuckDB Concurrency
------------------------
DuckDB supports only one writer at a time. This is a risk when:
  - The behavioral event API is writing rapidly
  - The analytics dashboard is reading simultaneously
  - The learning pipeline is running a batch update

  Mitigation strategy:
    - Use DuckDB's WAL (write-ahead log) mode
    - Write behavioral events to a fast append-only CSV buffer first;
      flush to DuckDB in micro-batches every 60 seconds
    - Schedule the learning pipeline to run during off-peak hours (2 AM)
    - Use read-only connections for the Shiny dashboard

1.5 Scraper Fragility and API Drift
--------------------------------------
LinkedIn's internal API endpoints are undocumented and change without notice.
The linkedin-api library may break when LinkedIn updates their mobile app API.

  Risks and mitigations:
    - Internal API endpoint changes → Pin the linkedin-api version in
      requirements.txt; monitor the library's GitHub issues for breakage
      reports; test scraper in CI against a mocked response fixture
    - Auth format changes (token structure, headers) → Library maintainers
      typically patch these within days; isolate auth logic in a single module
      so updates are contained to src/linkedin_project/scrape/auth.py
    - Field names or nesting changes in response JSON → Define an explicit
      schema contract in src/linkedin_project/scrape/schema.py; wrap all
      field access in a validation layer that raises descriptive errors:
      "Expected field 'timePeriod' in experience, got ['dateRange']"
    - Response completeness varies by account settings → Some fields
      (e.g., phone, address) may be absent if the account has privacy
      restrictions; treat all fields as optional with explicit nullability
    - Write schema-drift tests using fixture JSON files stored in
      tests/fixtures/linkedin_responses/ that fail fast on unexpected structure

2. DATA COLLECTION AND STRUCTURE
================================================================================

2.1 LinkedIn Profile Data (scraped via linkedin-api)
------------------------------------------------------
API call                        → Parsed into
--------                        → -----------
get_profile(public_id)          → owner name, headline, summary,
                                  location, industry, profile picture URL
get_profile_experiences()       → experience records (company, title,
                                  start, end, description, employment type)
get_profile_education()         → education records (school, degree,
                                  field of study, dates, activities)
get_profile_skills()            → skill names WITH endorsement counts
                                  (available via scraping, not CSV export)
get_profile_certifications()    → cert name, authority, issue date,
                                  expiry, credential URL
get_profile_recommendations()   → recommender name, relationship, text, date
get_profile_connections()       → connection count (aggregate only)
get_profile_contact_info()      → email, phone, websites, Twitter handle

Additional data available via scraping (not in CSV export):
  - Skill endorsement counts — enables endorsement-weighted skills chart
  - Connection count — usable as a social proof metric
  - Profile picture URL — for hero section of the resume
  - Company logo URLs from experience entries

All dates normalized to ISO 8601. All text fields stripped and UTF-8 encoded.
Raw scraped JSON responses cached to data/raw/ as timestamped files
(git-ignored) for debugging and re-processing without re-scraping.

2.2 Behavioral Event Data
--------------------------
Collected via JavaScript event listeners embedded in the Shiny app's UI,
posted to a Shiny reactive endpoint, persisted to DuckDB.

Events tracked:
  - page_view:      session start, referrer, device, viewport
  - section_view:   section entered/exited (IntersectionObserver), duration
  - viz_interact:   hover/click/expand on a specific visualization
  - viz_feedback:   explicit thumbs-up or thumbs-down on a visualization
  - scroll_depth:   25% / 50% / 75% / 100% scroll milestones per section
  - link_click:     outbound link clicked (e.g., GitHub, email)

2.3 Derived / Processed Data
------------------------------
Produced by the transform pipeline, stored in DuckDB, consumed by the app:

  Table: dim_experience        (normalized positions with computed tenure)
  Table: dim_education         (normalized education)
  Table: dim_skills            (deduplicated, categorized skills)
  Table: dim_certifications    (with active/expired flag)
  Table: fact_behavioral_events (raw event log)
  Table: agg_section_engagement (pre-aggregated engagement scores per section)
  Table: viz_suggestions       (AI-generated suggestions with status)

2.4 Data Refresh Cycle
-----------------------
  Profile data:    Automated scrape on schedule (weekly) OR triggered from
                   admin dashboard "Refresh Profile" button OR via CLI:
                   python scripts/scrape_linkedin.py
                   No manual ZIP download required.
  Behavioral data: Continuous append (event API → CSV buffer → DuckDB flush)
  Aggregations:    Scheduled daily at 2 AM via GitHub Actions cron job
  AI suggestions:  Scheduled weekly or triggered manually from admin dashboard

3. PRIVACY AND DATA SECURITY
================================================================================

3.1 Visitor Privacy
--------------------
Visitors to the resume are not asked to create accounts, but behavioral
signals are collected. This requires:

  - Cookie consent banner on first visit (GDPR/CCPA compliant)
    - "This site collects anonymous usage data to improve your experience."
    - Options: Accept / Decline (Decline disables all event tracking)
  - Session identifiers are hashed (SHA-256 of IP + user-agent + date salt)
    — never store raw IP addresses
  - No cross-site tracking; no third-party analytics pixels
  - Data retention: raw events purged after 90 days; aggregates retained indefinitely
  - Privacy policy page linked in footer

3.2 LinkedIn Credentials and Scraped Data
------------------------------------------
  Credentials:
    - LINKEDIN_USERNAME and LINKEDIN_PASSWORD stored as environment variables
    - Never hardcoded; never committed to git
    - Local: .env file (git-ignored)
    - CI/CD and production: GitHub Actions secrets / shinyapps.io env vars
    - If LinkedIn account has 2FA enabled, the linkedin-api library supports
      a LINKEDIN_2FA_SECRET env var for TOTP generation; document this in
      the project .env.example file

  Scraped data:
    - Raw JSON responses cached to data/raw/ (git-ignored) for debugging
    - Processed/clean data in DuckDB contains no credentials
    - Recommendations text contains third-party PII — consider whether to
      display full text or anonymize recommender names in the public resume
    - Profile picture is fetched by URL at render time; do not cache it
      in the repo or DuckDB

3.3 Secrets and Credentials
-----------------------------
All secrets managed via environment variables, never hardcoded:

  LINKEDIN_USERNAME         — LinkedIn account email for scraper auth
  LINKEDIN_PASSWORD         — LinkedIn account password for scraper auth
  LINKEDIN_2FA_SECRET       — (optional) TOTP secret if 2FA is enabled
  ANTHROPIC_API_KEY         — Claude API key for suggestion engine
  SHINYAPPS_TOKEN           — shinyapps.io deployment token
  SHINYAPPS_SECRET          — shinyapps.io deployment secret
  SHINYAPPS_ACCOUNT         — shinyapps.io account name
  DUCKDB_PATH               — path to DuckDB file (default: data/duckdb/resume.duckdb)
  EVENT_BUFFER_PATH         — path to CSV event buffer
  SUGGESTION_SCHEDULE_HOUR  — hour to run AI suggestion pipeline (default: 2)

  Local development: use a .env file (git-ignored)
  CI/CD: stored as GitHub Actions secrets
  Production: stored as shinyapps.io environment variables

3.4 Admin Dashboard Security
------------------------------
  - Admin dashboard (analytics + suggestions) accessible only via a password
    stored as an environment variable (ADMIN_PASSWORD)
  - No user authentication library required at this scale — a simple
    session token checked against a hash is sufficient
  - Admin routes clearly separated from public resume routes in the Shiny app

4. USER EXPERIENCE FLOW
================================================================================

4.1 Resume Visitor Flow
------------------------
1. Visitor arrives (via LinkedIn, email signature, portfolio link)
2. Consent banner appears — visitor accepts or declines tracking
3. Hero section loads: name, headline, current role, contact links
4. Visitor scrolls through sections:
     Summary → Experience (timeline viz) → Skills (categorized chart)
     → Education → Certifications → Projects → Recommendations
5. Each section has at least one interactive visualization
6. Visitor can optionally rate a visualization (thumbs up/down)
7. Outbound links (GitHub, email, LinkedIn) tracked as link_click events
8. On exit: session duration and final scroll depth recorded

4.2 Resume Owner (Admin) Flow
-------------------------------
1. Owner visits /admin route, enters ADMIN_PASSWORD
2. Dashboard displays:
     - Visitor analytics: sessions this week/month, top referrers, avg time
     - Section heatmap: which sections get the most attention
     - Visualization performance: engagement scores per viz
     - AI suggestions panel: pending suggestions with rationale
3. Owner reviews each pending suggestion:
     - Reads rationale and proposed visualization type
     - Accepts (suggestion queued for implementation) or rejects (archived)
4. Owner triggers LinkedIn re-scrape via "Refresh Profile" button
   (runs scraper → parses response → upserts DuckDB — no file download needed)
5. Owner can manually trigger AI suggestion pipeline

4.3 Data Scrape Flow (Owner, CLI or Admin Dashboard)
------------------------------------------------------
CLI:
1. Ensure LINKEDIN_USERNAME and LINKEDIN_PASSWORD are set in .env
2. Run: python scripts/scrape_linkedin.py
3. Script authenticates, fetches all profile sections, caches raw JSON
   to data/raw/linkedin_YYYY-MM-DD.json (git-ignored)
4. Script validates response against schema contract, transforms data,
   upserts DuckDB tables
5. Script prints summary: records upserted per section, any missing fields,
   scrape duration, and account status (no checkpoint detected / warning)
6. Shiny app reflects updated data on next page load

Admin Dashboard:
- Click "Refresh Profile" button → triggers same pipeline server-side
- Progress shown inline; errors surfaced with actionable messages

Scheduling (GitHub Actions cron):
- Runs weekly on Sunday at 1 AM UTC using stored GitHub secrets
- On auth failure, sends a GitHub Actions email notification to owner

5. EDGE CASES AND ERROR HANDLING
================================================================================

5.1 LinkedIn Scraping Errors
------------------------------
  Edge case                          → Handling
  ---------                          → --------
  Wrong credentials                  → Raise AuthError immediately with message:
                                       "LinkedIn login failed — check credentials"
  2FA checkpoint triggered           → Detect challenge response; if
                                       LINKEDIN_2FA_SECRET is set, generate
                                       TOTP code and submit; otherwise raise
                                       CheckpointError with instructions
  Account temporarily restricted     → Detect 999/redirect response; back off
                                       24 hours; log warning and notify owner
                                       via GitHub Actions email
  Rate limit / too many requests     → Exponential backoff (2s, 4s, 8s) between
                                       API calls; abort after 3 consecutive
                                       failures and surface clear error
  Field absent in API response       → All schema fields treated as Optional;
                                       log which fields were missing; continue
                                       processing with nulls
  Empty profile section              → Render graceful empty state in Shiny app;
                                       hide section from navigation
  Unexpected JSON structure           → Validation layer raises SchemaError:
                                       "Expected key 'timePeriod', got ['dateRange']"
  Duplicate records on re-scrape     → Upsert by natural key (company+title+start
                                       for experience; school+degree for education)
  Non-ASCII characters               → Normalize to UTF-8 on ingest
  Playwright fallback needed         → If linkedin-api raises 3 consecutive errors,
                                       automatically retry with Playwright scraper;
                                       log which path was used

5.2 Behavioral Event Collection
---------------------------------
  Edge case                          → Handling
  ---------                          → --------
  Visitor declines consent           → No events sent; all tracking code
                                       skipped client-side
  Event API unavailable              → Events queued in sessionStorage,
                                       retried on next interaction (max 3 retries)
  DuckDB write lock                  → Events written to CSV buffer; flushed
                                       asynchronously; never block the UI
  High traffic spike                 → CSV buffer absorbs load; DuckDB flush
                                       batches up to 1000 events at a time
  Bot traffic                        → Filter events where session_duration < 2s
                                       and scroll_depth = 0% during aggregation

5.3 AI Suggestion Pipeline
----------------------------
  Edge case                          → Handling
  ---------                          → --------
  Claude API unavailable             → Retry 3x with exponential backoff;
                                       if all fail, skip cycle and log error
  Claude returns malformed JSON      → Parse with error recovery; if unparseable,
                                       discard response and log raw output
  Insufficient behavioral data       → Skip AI call; activate rule-based
                                       seed suggestions (cold start mode)
  Suggestion for unsupported viz     → Filter suggestions to known viz types
                                       supported by the Shiny/matplotlib stack
  Owner never reviews suggestions    → Suggestions expire after 30 days;
                                       archived automatically

5.4 Application Runtime
-------------------------
  Edge case                          → Handling
  ---------                          → --------
  DuckDB file missing on startup     → App initializes empty DB with schema;
                                       shows "Import your LinkedIn data to begin"
  Missing environment variable       → Fail fast on startup with clear error:
                                       "Required env var X is not set"
  shinyapps.io memory limit          → Profile memory usage; lazy-load heavy
                                       visualizations only when section scrolled
                                       into view
  Profile section completely empty   → Hide section from navigation and page
                                       rather than rendering empty card

6. DEPLOYMENT AND MAINTENANCE STRATEGY
================================================================================

6.1 Deployment Architecture
-----------------------------

  Component              → Host
  ---------              → ----
  Python Shiny App       → shinyapps.io (free tier initially; scale as needed)
  DuckDB database        → Bundled with app deployment (file-based)
  GitHub Actions CI/CD   → github.com/chrisselig/linkedin_resume_ai
  AI suggestion pipeline → GitHub Actions scheduled workflow (cron)

  Note on DuckDB on shinyapps.io:
    shinyapps.io does not provide persistent storage between deployments.
    The DuckDB file must be:
    - Committed to the repo (only the processed/clean DB, no raw PII)
    - OR fetched from a remote source on startup (e.g., GitHub Release asset,
      or a private S3-compatible bucket)
    Recommended approach: commit a minimal seed DB; re-import profile data
    via the admin dashboard after each deployment.

6.2 CI/CD Pipeline
-------------------
  Trigger: Pull Request to main
    - black --check .
    - flake8
    - pytest --cov=src --cov-report=term-missing
    - Coverage threshold: 80% minimum

  Trigger: Push to main
    - Run full CI suite
    - Deploy to shinyapps.io via rsconnect-python
    - Post deployment URL as commit status

  Trigger: Scheduled (daily 2 AM UTC)
    - Run behavioral data aggregation pipeline
    - Flush any pending CSV buffer events to DuckDB
    - Update agg_section_engagement table

  Trigger: Scheduled (weekly Sunday 3 AM UTC)
    - Run AI suggestion pipeline
    - Write new suggestions to viz_suggestions table

6.3 Maintenance Strategy
--------------------------

  Scraper Maintenance:
    - Monitor linkedin-api GitHub repo for breakage reports and new releases
    - Pin library version in requirements.txt; test upgrades in a dedicated PR
    - Run schema validation tests against fixture JSON files stored in
      tests/fixtures/linkedin_responses/ (synthetic/anonymized data)
    - When LinkedIn changes internal API structure, update schema.py and
      the response parser; update fixture files to match new structure
    - If the library breaks and no patch is available within a week,
      activate the Playwright fallback path as temporary replacement

  Dependency Management:
    - Pin all dependencies in requirements.txt
    - Run pip-audit weekly (add to scheduled CI workflow) to detect
      known vulnerabilities
    - Upgrade dependencies in a dedicated PR; re-run full test suite

  AI Suggestion Quality:
    - Track acceptance rate of suggestions (accepted / total)
    - If acceptance rate drops below 30% over 4 weeks, review and
      refine the Claude API prompt in src/linkedin_project/suggestions/prompt.py
    - Log all API calls and responses for debugging

  Monitoring:
    - shinyapps.io provides basic uptime and usage metrics
    - Implement a /health endpoint in the Shiny app that returns:
      {"status": "ok", "db_connected": bool, "last_import": ISO_date}
    - Add an uptime check (e.g., UptimeRobot free tier) that pings /health

6.4 Scaling Considerations
----------------------------
  The current architecture is intentionally sized for a personal resume
  (low traffic, single owner). If usage grows significantly:

  - DuckDB → MotherDuck (serverless DuckDB cloud) for persistent storage
    without managing file deployments
  - shinyapps.io → Posit Connect (self-hosted) or a containerized deployment
    (Docker + Fly.io / Railway) for more control and persistent storage
  - Behavioral event buffer → Replace CSV buffer with a lightweight message
    queue (Redis Streams or a free-tier managed Kafka)
  - AI suggestions → Add a vector store (e.g., ChromaDB) to enable
    semantic search over past suggestions and avoid duplicate proposals

================================================================================

  1. LinkedIn scraper (linkedin-api auth + all profile API calls + Playwright fallback)

  10. Claude API integration for suggestion generation
  11. Suggestion review UI in admin dashboard
  12. Cold-start rule-based seed suggestions

  13. Consent banner + privacy policy
  14. Error handling audit across all modules
  15. Performance profiling (memory, load time on shinyapps.io)
  16. Documentation update and README finalization

8. KEY DESIGN DECISIONS SUMMARY
================================================================================

Decision                    Choice                    Rationale
--------                    ------                    ---------
LinkedIn data source        linkedin-api (unofficial  Automated, schedulable,
                            internal API) +           returns richer data than
                            Playwright fallback       CSV export (endorsement
                                                      counts, profile picture);
                                                      ToS risk accepted for
                                                      personal/own-profile use
Database                    DuckDB                    Zero-infra, analytical
                                                      queries, pandas integration
Event buffer                CSV append + batch flush  Avoids DuckDB write lock;
                                                      simple and reliable
AI layer                    Claude API (claude-       Best-in-class for structured
                            sonnet-4-6)               output; controllable prompts
Learning mechanism          Engagement scoring +      Transparent, debuggable,
                            LLM suggestion            no black-box ML needed
Auth (admin)                Single password + hash    Appropriate for single-owner
                                                      app; no auth library bloat
Deployment                  shinyapps.io              Zero-infra; native Shiny
                                                      support; free tier sufficient
Storage on shinyapps.io     Committed seed DB +       Pragmatic for current scale;
                            admin re-import           migration path to MotherDuck
